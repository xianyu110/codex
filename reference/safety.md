# 避坑与安全

Codex 的价值在于能执行任务：读文件、改文件、运行命令、控制浏览器、连接外部系统。也正因为它能执行，越要有边界。

## 一句话原则

让 Codex 做事前，先确认三件事：

1. 它会读什么。
2. 它会改什么。
3. 它会把什么发到外部。

## 绝对不要做

- 不要把 API Key、token、cookie、私钥发给 Codex。
- 不要把 `.env` 内容贴到聊天里。
- 不要让 Codex 把密钥写进 README、日志、截图或代码。
- 不要在陌生项目里直接 Full Access 跑安装脚本。
- 不要让 Codex 自动处理支付、转账、删除生产数据。
- 不要把第三方中转 Key 提交到 GitHub。
- 不要把浏览器登录态、验证码、后台页面截图随意发给不需要的任务。

## 密钥应该怎么处理

错误做法：

```text
这是我的 API Key：sk-xxxx，请写进代码里。
```

正确做法：

```text
请把代码改成从环境变量读取 API Key。告诉我需要设置的环境变量名和本地运行命令，不要把密钥写进代码或日志。
```

代码里应该类似这样：

```js
const apiKey = process.env.OPENAI_API_KEY;
if (!apiKey) {
  throw new Error('OPENAI_API_KEY is required');
}
```

## 权限风险表

| 权限 | 风险 | 建议 |
| --- | --- | --- |
| Read only | 低 | 审查、解释、学习首选 |
| Workspace write | 中 | 日常开发默认 |
| Network enabled | 中 | 只在查文档、安装依赖、部署时开 |
| Full Access | 高 | 跨目录、本机运维才用 |
| Browser / Chrome 控制 | 中高 | 不要让它操作支付、隐私、后台敏感页 |
| Computer Use | 高 | 会影响系统和 App 状态，任务要非常明确 |
| MCP 写操作 | 高 | 发消息、改 issue、写数据库前确认 |

## 高风险提示词改写

| 不要这样说 | 改成这样说 |
| --- | --- |
| 你直接执行所有命令，不用问我 | 先列命令和用途，危险命令等我确认 |
| 把这个 token 写进代码 | 改成读取环境变量 |
| 帮我清理没用文件 | 先列出候选文件和删除理由 |
| 直接重置仓库 | 先说明会丢失哪些改动 |
| 自动发布到所有平台 | 先生成草稿，发布前人工确认 |
| 连接数据库随便查 | 只读查询，限制表和字段 |

## 第三方中转说明

本教程保留的国内站和第三方中转入口属于第三方接入方案，不是 OpenAI 官方订阅。

保留链接：

- `https://codex.chatgpt-plus.top/login`
- `https://codex2.chatgpt-plus.top/login`
- `https://maynorai.jichiyun.sbs/buy/30`
- `https://maynorai.jichiyun.sbs/buy/13`
- `https://maynorai.jichiyun.sbs/buy/7`

使用第三方中转时要注意：

- 明确自己正在使用第三方服务。
- API Key 不写入仓库。
- 不在公开 issue、README、截图中暴露 Key。
- 不把中转服务描述成 OpenAI 官方订阅。
- 出错排查时只提供错误类型和请求路径，不提供完整密钥。

## 安装依赖前检查

让 Codex 安装依赖前，先让它回答：

```text
请先说明准备安装什么包、为什么需要、是否会修改 lockfile、有没有更轻量的替代方案。
```

检查点：

- 这个依赖是否真的需要。
- 是否是项目现有技术栈的一部分。
- 是否会引入网络请求或遥测。
- 是否会修改大量文件。
- 是否能用标准库或已有依赖解决。

## 删除和回退前检查

高风险命令包括：

```bash
rm -rf
git reset --hard
git checkout -- .
git clean -fd
```

让 Codex 执行前，必须确认：

- 会删除哪些文件。
- 是否包含未提交改动。
- 是否有备份或 Git 记录。
- 是否可以先用 `git status` 和 `git diff` 查看。

## 浏览器和 Computer Use 安全

适合：

- 本地网页预览。
- 公开页面检查。
- 表单布局测试。
- 桌面 App 的非敏感操作。

不适合自动执行：

- 支付。
- 转账。
- 删除账号或数据。
- 批量发消息。
- 输入验证码。
- 操作后台生产配置。

如果必须操作登录态页面，提示词要限定范围：

```text
只查看页面结构并截图，不点击删除、提交、支付、发布、发送等按钮。
```

## MCP 安全清单

启用 MCP 前检查：

- 这个 MCP 能读哪些数据？
- 能不能写入或删除？
- 是否会发送消息？
- 是否会调用外部 API？
- token 从哪里来？
- 是否有日志？
- 是否能限制工具列表？

对数据库 MCP 的默认建议：

- 先只读。
- 限制数据库、schema、表。
- 不查敏感字段。
- 不导出全量数据。
- 写操作必须人工确认。

## 提交前安全检查

每次任务结束运行或让 Codex 运行：

```bash
git diff --check
git status --short
git grep -n "sk-[A-Za-z0-9]" -- . ':!*.png' ':!*.jpg' ':!*.jpeg' || true
```

再人工确认：

- diff 里没有密钥。
- 日志里没有密钥。
- 没有新增不必要网络请求。
- 没有新增统计、遥测或上报代码。
- 删除和覆盖操作有明确理由。
- 保留链接没有被误删。

## 出问题后怎么处理

如果 Codex 改坏了：

1. 先停下，不要继续让它扩大修改。
2. 看 `git status --short`。
3. 看 `git diff`。
4. 指出具体文件和错误。
5. 让 Codex 做最小修复。

可用提示词：

```text
你刚才的改动有问题。请只针对当前 diff 做最小修复，不要重写整个文件。
问题是：[具体问题]。
完成后重新运行刚才失败的检查。
```
