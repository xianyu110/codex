# Codex 完整实战教程: 从安装到第一个可交付项目

这是一条给新手准备的完整路线。你可以把它当成一节长课来读：先把 Codex 跑起来，再学会怎么给任务、怎么控制权限、怎么检查结果，最后做出一个能打开、能验证、能发布的静态网页。

如果你只想查某个功能，可以回到 [教程目录](../index.md) 按章节阅读。

## 这门课最终要做到什么

学完以后，你应该能独立完成这 5 件事：

1. 安装并打开 Codex App，或在终端启动 Codex CLI。
2. 分清官方订阅、第三方中转 API、桌面 App、CLI 之间的关系。
3. 给 Codex 一个清楚的任务，并让它先规划、再修改、再验证。
4. 使用 `AGENTS.md`、Skills、MCP、自动化，把一次经验沉淀成可复用流程。
5. 完成第一个实战项目：把 README 整理成网页，并确认原来的国内站和购买链接没有丢。

## 0. 先建立正确心智

Codex 不是一个只负责聊天的问答框。它更接近一个能读文件、改文件、运行命令、检查结果、继续修正的 coding agent。

新手最容易踩的坑，是把任务写成一句模糊愿望：

```text
帮我优化一下这个项目。
```

更好的写法是：

```text
目标：把这个仓库改造成 Codex 中文实战教程站。
上下文：README 里有原来的国内站、备用入口和购买链接。
约束：不要删除原有中转站、购买、飞书教程链接；不要新增统计代码。
完成标准：docs/guide 有入门教程，recipes 有案例库，首页能进入教程和案例，并跑本地预览检查。
```

记住这个公式：**目标 + 上下文 + 约束 + 完成标准**。这比单纯追求“更强模型”更重要。

## 1. 选择入口: App、CLI、Web、IDE

Codex 有多个入口。第一天不要全都装，先选一个跑通。

| 入口 | 适合谁 | 你会看到什么 | 建议 |
| --- | --- | --- | --- |
| 桌面 App | 小白、内容创作者、需要可视化的人 | Thread、计划、文件改动、终端、Diff | 本教程默认推荐 |
| CLI | 开发者、经常在终端工作的人 | 终端 TUI、命令、diff、slash commands | 熟悉命令行再用 |
| Web / Cloud | 想把任务放到云端跑的人 | 远程任务、PR、review | 适合团队和并行任务 |
| IDE 扩展 | 已经在 VS Code / Cursor 里写代码的人 | 编辑器上下文和 Codex 线程 | 适合真实开发 |

如果你完全不知道选哪个，先用桌面 App。它更容易看到 Codex 正在做什么，也更容易在它改错时停下来。

## 2. 安装 Codex App

最稳的路径是先从官方入口了解当前版本，再使用本教程保留的下载入口。

1. 打开官方入口：`https://chatgpt.com/codex`
2. 下载适合你系统的 Codex App。
3. 安装并登录 ChatGPT 账号。
4. 添加一个本地项目目录。
5. 新建 Thread，先让 Codex 只读分析项目。

本仓库原来保留的下载入口继续保留：

- macOS 安装包：`https://persistent.oaistatic.com/codex-app-prod/Codex.dmg`
- Windows 本地安装包：`https://pan.quark.cn/s/0f1763fe2ac9?pwd=rTQw`

第一次打开后，不要马上让它大改项目。先发这句：

```text
请先只读检查这个项目，告诉我主要文件、运行方式和你建议的第一个小任务。暂时不要修改文件。
```

完成标准：

- Codex 能看到你的项目目录。
- 它能识别 README、入口文件、配置文件。
- 它没有在你要求只读时修改文件。

## 3. 订阅和中转 API 怎么选

这里要分清三件事：

1. **官方订阅**：ChatGPT Plus / Pro / Business 等账号套餐。
2. **第三方中转 API**：降低访问、支付、配置门槛的第三方接入方案。
3. **客户端**：Codex App、CLI、IDE、Web 只是使用入口。

官方订阅适合能稳定访问 ChatGPT、希望使用官方原生体验的人。第三方中转适合希望更快跑通、不想处理网络和支付细节的人。

| 备用入口 | `https://codex2.chatgpt-plus.top/login` |
| Codex&GPTimage2 套餐 | `https://maynorai.jichiyun.sbs/buy/30` |
| 配置中转 API | `https://maynorai.jichiyun.sbs/buy/13` |
| 充值/订阅相关入口 | `https://maynorai.jichiyun.sbs/buy/7` |

## 4. App 界面怎么用

你可以把 Codex App 看成一个“任务工作台”。

| 区域 | 作用 | 新手怎么用 |
| --- | --- | --- |
| Projects | 管理本地项目 | 一个仓库一个项目，不要一开始加整个硬盘 |
| Threads | 管理任务上下文 | 一条 Thread 只做一件事 |
| Composer | 输入任务 | 写目标、上下文、约束、完成标准 |
| Plan Mode | 先规划再执行 | 多文件任务先打开 |
| Terminal | 运行命令和看输出 | 让 Codex 解释命令用途 |
| Diff / Changes | 查看改动 | 提交前必须看 |
| Settings | 设置权限、MCP、外观、个性化 | 先保持保守配置 |

一个很好的第一次任务是：

```text
请阅读当前项目，列出：
1. 这个项目是什么；
2. 主要目录和文件；
3. 如何本地预览；
4. 你建议先补哪一部分教程。
暂时不要修改文件。
```

如果 Codex 给出的文件名不对，先纠正它，不要急着让它改。

## 5. 权限与沙盒: 默认不要 Full Access

Codex 能运行命令，所以权限要认真选。

| 模式 | 适合场景 | 风险 |
| --- | --- | --- |
| Read only | 解释项目、做 review、读文档 | 不能直接改文件 |
| Workspace write | 日常开发、改教程、补页面 | 仍要注意命令和密钥 |
| Full Access / danger-full-access | 跨目录整理、系统配置、本机运维 | 权限很大，只给可信任务 |
| Network enabled | 查文档、安装依赖、部署 | 要警惕网页内容和依赖脚本 |

新手默认用 Workspace write。陌生仓库第一次打开时，先用只读提示词：

```text
先只读检查项目结构和脚本。请列出你准备运行的命令、每个命令的用途，以及是否需要网络。不要读取 .env，不要打印任何密钥。
```

Full Access 只在你清楚知道为什么需要时使用。比如跨目录找素材、整理本机文件、部署到外部服务。即使开了高权限，也要让 Codex 先说明要访问哪些路径。

## 6. 学会 Plan Mode

复杂任务先计划，不要直接开改。

适合 Plan Mode 的任务：

- 改造一个仓库。
- 多个文件同时变化。
- 可能影响构建或部署。
- 有必须保留的链接、样式、配置。
- 你还不确定该怎么做。

可以这样写：

```text
/plan
目标：把这个仓库做成完整 Codex 实战教程。
请先检查目录、README、首页、docs、recipes、reference。
约束：保留原来的国内站、备用入口、购买入口和飞书教程链接。
输出：给我 3-6 条计划，说明要改哪些文件、怎么验证，先不要改文件。
```

看计划时重点检查：

- 它是否识别了正确仓库。
- 它是否知道哪些链接不能删。
- 它有没有验证步骤。
- 它有没有把范围扩大到无关重构。

## 7. 斜杠命令速记

不同 Codex 入口的命令会有差异，以当前界面 `/help` 为准。新手先记这些：

| 命令 | 作用 | 常见用法 |
| --- | --- | --- |
| `/plan` | 进入规划模式 | 大任务开始前 |
| `/status` | 查看线程、上下文、限制、配置 | 长任务中途 |
| `/help` | 查看当前可用命令 | 不确定时 |
| `/model` | 查看或切换模型 | 需要更强或更快时 |
| `/permissions` | 调整权限 | CLI 中常用 |
| `/review` | 做代码 review | 提交前或 PR 前 |
| `/mcp` | 查看 MCP 状态 | 工具连不上时 |
| `/init` | 生成 AGENTS.md 草稿 | 新仓库初始化 |

一个常用组合：

```text
/plan
请先规划这个任务。

/status
查看当前状态。

请按计划执行第一步，并在改文件前说明要改哪些文件。
```

## 8. 推理深度怎么选

不要所有任务都开最高。推理深度越高，通常越适合复杂任务，但也会更慢、更耗额度。

| 任务 | 推荐 |
| --- | --- |
| 改错别字、补链接、改 Markdown | Low / Medium |
| 单文件小改、简单测试 | Medium |
| 多文件功能、真实 bug、排查失败构建 | High |
| 架构迁移、复杂链路、长期目标 | Extra High |

更重要的是提示词清楚。坏提示词配 Extra High 也可能乱跑；好提示词用 Medium 也能稳定完成。

## 9. Threads 怎么组织

Thread 是任务容器。一条 Thread 尽量只做一件事。

推荐：

- `docs: 补齐 Codex 入门教程`
- `recipes: 新增 Playwright MCP 案例`
- `fix: 修复首页移动端按钮换行`

不推荐：

- `帮我把整个项目都优化一下`
- `顺便把部署、SEO、公众号文章和封面都做了`

并行时要注意：多个 Thread 不要同时改同一个文件。如果你想并行，可以这样拆：

- Thread A：补 `docs/guide`。
- Thread B：补 `recipes`。
- Thread C：检查链接和本地渲染。

每条 Thread 结束时，让 Codex 回答：

```text
请总结：目标是否完成、改了哪些文件、跑了哪些检查、还有什么风险。
```

## 10. AGENTS.md: 把规则写进仓库

`AGENTS.md` 是写给 Codex 的项目说明。它适合放长期规则，不适合放一次性任务。

这个教程站可以使用这样的模板：

```md
# AGENTS.md

## Project

这是 Codex 中文实战教程站，内容以 Markdown 和静态 HTML 为主。

## Commands

- 本地预览：python3 -m http.server 8765
- 搜索内容：rg "关键词"
- 检查 Git 状态：git status --short

## Rules

- 默认使用中文。
- 保留原有国内站、备用网址、购买入口、飞书教程链接。
- 不要新增统计、遥测或不必要网络请求。
- 不要把 API Key、token、cookie 写入代码、文档或日志。
- 改动后说明文件列表和验证方式。

## Done

- 新页面能从目录进入。
- Markdown 链接使用相对路径。
- 本地预览能打开关键页面。
```

什么时候更新 `AGENTS.md`？

- Codex 重复犯同一个错。
- 项目有固定命令和目录约定。
- PR review 经常重复同一条反馈。
- 某个子目录有特殊规则。

## 11. Skills: 把重复流程做成能力

Skill 适合沉淀可复用流程。它通常是一个目录，里面有 `SKILL.md`，也可以带模板、脚本、示例。

适合做成 Skill 的任务：

- 公众号文章整理。
- PPT 生成。
- PDF 审阅。
- 把 README 变网页。
- 某个固定项目的发布流程。

一个最小 `SKILL.md` 可以这样写：

```md
---
name: readme-to-web
description: 把 README 整理成静态网页。用于用户要求把文档改造成首页、教程页或发布页时。
---

1. 先搜索 README 中所有外部链接，列出必须保留的链接。
2. 读取现有首页和样式，沿用当前设计风格。
3. 首页只保留入口、学习路线和关键 CTA。
4. 长教程移到 docs/guide，案例移到 recipes。
5. 完成后运行本地预览并检查关键链接。
```

Skill 的核心不是“写一大堆提示词”，而是把触发条件、输入、步骤、输出和验证标准写清楚。

## 12. MCP: 连接外部系统

MCP 用来让 Codex 访问外部工具或数据源。比如 GitHub、Figma、Playwright、数据库、Notion、飞书等。

什么时候用 MCP？

| 场景 | 该不该用 MCP |
| --- | --- |
| 只改本地 Markdown | 不需要 |
| 读取 Figma 设计稿 | 需要 |
| 控制浏览器测试页面 | 需要 |
| 管理 GitHub issue / PR | 可以 |
| 查询数据库 | 可以，但要控制权限 |
| 访问私有业务系统 | 可以，但要先确认授权 |

配置 MCP 时要关注三件事：

1. 它能读什么。
2. 它能写什么。
3. 哪些操作需要人工确认。

如果 MCP 要用 token，优先通过环境变量传入，不要写进仓库。

## 13. 自动化: 只自动化已经跑通的流程

自动化适合稳定、重复、可验证的任务，不适合一开始就拿来探索需求。

适合：

- 每天检查网页更新。
- 定时生成日报。
- 定时检查 CI 失败。
- 周期性整理知识库。
- 到点回到某个 Thread 继续跟进。

不适合：

- 需求每天变。
- 每次都要人工判断。
- 涉及支付、删除、生产数据。
- 没有日志，失败无法定位。

设计自动化时，用这个清单：

| 项目 | 必须写清楚 |
| --- | --- |
| 触发 | 每天几点、每小时、还是手动 |
| 输入 | URL、仓库、文件夹、关键词 |
| 去重 | 如何避免重复处理 |
| 输出 | Markdown、PR、消息、表格 |
| 日志 | 成功/失败写到哪里 |
| 人工确认 | 哪些动作不能自动做 |

推荐提示词：

```text
请把这个已经跑通的流程设计成 Codex 自动化。
要求：只处理新增内容；保留 state.json 去重；失败写入 logs/errors.jsonl；
每次汇报扫描数量、新增数量、成功数量、失败数量和日志路径。
```

## 14. CLI 第一次使用

如果你熟悉终端，可以用 CLI。

```bash
cd /path/to/project
codex
```

进入后先只读：

```text
请阅读这个仓库，找出入口文件、测试命令和构建命令。暂时不要改文件。
```

常用命令：

```bash
git status --short
git diff
rg "关键词"
python3 -m http.server 8765
```

CLI 中常用斜杠命令包括 `/plan`、`/status`、`/model`、`/permissions`、`/review`、`/mcp`、`/init`。如果不确定当前版本支持什么，直接输入 `/help`。

## 15. 第一个可验证任务: README 变网页

这是最适合新手的第一个实战，因为它能练到完整闭环：

- 读文件。
- 找关键链接。
- 改 HTML / Markdown。
- 本地预览。
- 检查 diff。
- 修复问题。
- 提交推送。

给 Codex 的提示词：

```text
请把 README.md 的核心内容整理成一个静态首页。
要求：
1. 保留原来的国内站、备用入口、购买套餐、飞书教程链接。
2. 首页不要太长，只保留入口、学习路线和实战案例入口。
3. 长教程放到 docs/guide，案例放到 recipes。
4. 完成后用本地预览检查页面是否能打开。
5. 汇报改动文件、验证命令和保留的关键链接。
```

本地验证：

```bash
python3 -m http.server 8765
```

打开：

```text
http://127.0.0.1:8765/
```

检查：

- 首页能打开。
- `docs/`、`recipes/`、`reference/` 能进入。
- 国内站入口还在。
- 备用网址还在。
- 购买入口还在。
- 飞书教程链接还在。
- 移动端没有明显遮挡和横向溢出。

## 16. 一套完整交付流程

当 Codex 完成修改后，不要只听它说“完成了”。你要让它给证据。

推荐收尾提示词：

```text
请做收尾检查：
1. git status --short；
2. git diff --check；
3. 检查所有必须保留的中转/购买/飞书链接；
4. 本地预览首页、docs、recipes、reference、reader 页面；
5. 总结改动文件、验证结果和剩余风险。
```

如果要上传 GitHub：

```text
请提交并推送到当前仓库。提交前确认 remote 是正确仓库，工作树只包含本次改动。
```

一个合格的最终汇报应该包含：

- 推送到哪个仓库。
- 最新 commit hash。
- 改了哪些文件。
- 跑了哪些检查。
- 原有链接是否保留。
- 还有什么没做。

## 17. 常见问题

**Q: 我不会写代码，能用 Codex 吗？**

可以。先从文档整理、网页生成、PPT、Obsidian、资料调研开始。这些任务同样能训练你给目标、约束和完成标准的能力。

**Q: Codex 改错了怎么办？**

先看 diff，指出具体文件和问题。不要说“你改错了”，而是说“`docs/index.md` 里把购买链接删了，请恢复并重新检查所有保留链接”。

**Q: 什么时候需要 MCP？**

当任务需要外部系统数据或操作时，比如 Figma、GitHub issue、浏览器、数据库。只改本地文件通常不需要。

**Q: 我应该把 API Key 发给 Codex 吗？**

不要。让 Codex 把代码改成从环境变量读取，并告诉你环境变量名。

**Q: 怎么判断教程站算完成？**

至少满足：主线教程能从首页进入；15 个入门章节齐全；案例库有分类和可执行提示词；速查有命令、设置、安全、FAQ；原来的国内站和购买入口没有丢；本地预览能打开关键页面。
